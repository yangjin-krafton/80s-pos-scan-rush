# POS SCAN RUSH

1995년 도쿄 편의점 계산대를 배경으로 한 세로형 브라우저 미니게임 프로토타입입니다. 플레이어는 카트에서 상품 카드를 집어 스캐너로 드래그하고, 할인율 또는 무료 여부를 맞춘 뒤 POS 계산을 완료해야 합니다. 손님 만족도가 먼저 바닥나면 실패합니다.

현재 저장소 기준으로 이 프로젝트는 "80년대 마트 계산대 시뮬레이션 기획서"보다는, 360x640 모바일 화면에 맞춘 정적 웹게임 프로토타입에 가깝습니다. `package.json` 없이 `src/`를 그대로 서빙하는 구조이며, 핵심은 손맛 있는 홀드 스캔, 점진적으로 해금되는 튜토리얼, 절차 생성 라운드, 레트로 UI 연출입니다.

## 이 프로토타입이 실제로 된 것

- 바닐라 JS/CSS 기반의 정적 브라우저 게임
- Win95 창 스타일 POS + 픽셀풍 손님 씬 + 카트 카드 드래그 UI
- `products.csv`, `npcs.json`, `promos.json`를 읽어 라운드를 절차 생성
- 난이도(`diffRating`)에 따라 새 기믹이 풀리는 끝없는 라운드 구조
- 첫 플레이용 기본 튜토리얼과 기믹별 튜토리얼/연습 라운드
- WebAudio 기반 SFX/BGM, 파티클, 화면 이펙트, 볼륨 설정 패널

게임 진행은 대체로 아래 흐름입니다.

1. 타이틀 오버레이를 넘기고 게임을 시작합니다.
2. 하단 카트에서 상품 카드를 드래그해 스캐너 위에 올립니다.
3. 할인 상품이면 할인 버튼으로 할인율을 먼저 맞추고, 프로모션 상품이면 무료 토글을 켭니다.
4. 바코드가 스캐너 영역에 일정 시간 겹치면 스캔 성공으로 POS에 등록됩니다.
5. 모든 항목과 수량을 맞춘 뒤 `계산`을 눌러 다음 손님으로 넘어갑니다.

## 현재 구현된 핵심 기믹

| 구분 | 현재 구현 |
| --- | --- |
| 기본 스캔 | 바코드 겹침을 일정 시간 유지해야 성공하는 홀드 스캔 |
| 만족도/점수 | 시간 경과로 만족도 감소, 성공 스캔과 계산으로 점수 획득, 실수 시 감점 |
| 할인 상품 | 할인율 버튼으로 맞는 할인 바코드를 선택해야 정답 처리 |
| 수량 보정 | POS의 `+/-` 버튼으로 수량 수정 가능 |
| 훼손 바코드 | 일부 복사본은 스캔이 안 되며 POS 수량 보정으로 해결 |
| 프로모션 | `1+1`, `2+1` 프로모션과 무료 상품 스캔 토글 |
| 실시간 이벤트 | 플레이 도중 추가 상품 요청, 취소 요청, POS 정전 |
| 변형 라운드 | 대량 구매, 같은 상품의 세일/비세일 혼합, 같은 상품의 복수 할인율 |
| 튜토리얼 | 기본 튜토리얼 + 기믹별 tutorial/practice 라운드 삽입 |
| 디버그 | 콘솔에서 난이도 고정, 갓모드, 메타 이벤트 강제 발동 |

## AI/자동화 사용 지점

현재 코드베이스에는 게임 플레이 중 모델을 호출하는 AI/LLM 기능이 없습니다. 에이전트, 프롬프트, 메모리, 모델 선택 같은 런타임 AI 구조도 들어 있지 않습니다.

대신 저장소에 남아 있는 자동화는 아래 쪽입니다.

- `src/sound/*.py`: 배포용 SFX/BGM 자산을 로컬에서 다시 생성하는 스크립트
- `sandbox/concept-multimon/`: UI 목업용 HTML 샌드박스
- `sandbox/bgm-mixer/`: BGM 스템 밸런스 확인용 HTML 샌드박스

즉, 이 저장소에서 확인되는 자동화는 "프로토타입 제작 보조 도구"에 가깝고, "게임 안에서 판단하는 AI"는 아닙니다.

## 무엇을 시도했고, 결과가 어땠는가

### 잘 작동한 쪽

- 드래그 후 짧게 홀드하는 스캔 루프가 단순 클릭보다 훨씬 게임답게 느껴지도록 정리됐습니다.
- 고정 스테이지 대신 절차 생성 라운드와 난이도 적응 로직이 붙으면서 반복 플레이 구조가 생겼습니다.
- 할인, 훼손 바코드, 프로모션, 추가/취소 요청 같은 예외 상황이 튜토리얼 해금과 연결되어 학습 흐름이 자연스러워졌습니다.
- 사운드와 오버레이, 큐 미리보기, 계절 이펙트 등으로 레트로 연출의 일관성이 좋아졌습니다.

### 단순화되거나 빠진 쪽

- 초기 기획의 좌측 "비밀 봉투" 수동 담기 흐름은 현재 없습니다. 스캔 성공 후 일정 시간 뒤 자동으로 정리됩니다.
- 캠페인형 `01/10` 진행보다, 실제 코드는 `endlessRounds: true` 기준의 무한 진행 구조에 가깝습니다.
- 일일 규칙, 저장/불러오기, 메타 진행도 저장, 별도 배포 파이프라인은 아직 없습니다.
- 아트 자산은 최종 게임용보다는 이모지와 CSS 중심의 프로토타입 표현에 머물러 있습니다.

## 초기 계획과 지금 구현의 차이

| 항목 | 초기 README의 방향 | 현재 구현 |
| --- | --- | --- |
| 게임 구조 | 10라운드 내외의 정답형 결제 게임 | 적응형 난이도로 계속 이어지는 라운드 스트림 |
| 조작 흐름 | 클릭으로 중앙 이동 후 스캔, 봉투 담기 | 카드를 직접 드래그해서 스캔, 스캔 후 자동 정리 |
| 핵심 예외 | 할인 스티커 위주 | 할인 + 훼손 바코드 + 프로모션 + 중간 요청/취소 + POS 정전 |
| 학습 방식 | 문서상 단계 설명 중심 | 인게임 툴팁 + 기믹별 tutorial/practice 라운드 |
| 문서 초점 | 구현 예정 체크리스트 | 이미 돌아가는 프로토타입의 범위와 한계 |

이 차이는 중요합니다. 지금 저장소를 보는 사람은 "원래 만들고 싶었던 것"보다 "현재 무엇을 직접 실행하고 테스트할 수 있는가"를 먼저 알아야 하기 때문입니다.

## 현재 범위

- 상품 데이터 100종 (`src/data/products.csv`)
- NPC 데이터 50종 (`src/data/npcs.json`)
- 응원 대사 22종 (`src/data/encouragements.json`)
- 프로모션 규칙 2종 (`1+1`, `2+1`)
- 기본 게임 진입용 타이틀/로딩/오버레이 UI
- 모바일 세로 화면 기준 스케일링
- 브라우저 콘솔 디버그 명령 지원

## 한계

- 저장 기능이 없어 새로고침 시 진행 상태가 유지되지 않습니다.
- 파일만 더블클릭해서 열면 `fetch()` 기반 데이터 로딩 때문에 정상 동작하지 않습니다. 로컬 서버가 필요합니다.
- 브라우저에서의 최종 플레이 감각은 오디오 권한, 터치 환경, 성능에 영향을 받습니다.
- 현재 턴에서는 브라우저 상호작용까지 재검증하지는 못했고, 정적 서버로 `index.html`과 데이터 파일이 응답하는 것까지만 확인했습니다.

## 실행 방법

```bash
python3 -m http.server 4173 --directory src
```

브라우저에서 아래 주소를 열면 됩니다.

```text
http://127.0.0.1:4173/index.html
```

참고:

- `file://`로 열면 데이터 로딩이 실패합니다.
- 별도 설치 스크립트나 빌드 단계는 없습니다.

## 디버그/실험 도구

- 디버그 명령어: `docs/debug-commands.md`
- UI 실험: `sandbox/concept-multimon/index.html`
- BGM 밸런스 실험: `sandbox/bgm-mixer/index.html`

브라우저 콘솔에서 자주 쓰는 명령은 아래 정도입니다.

```js
POS.debug.help()
POS.debug.god()
POS.debug.setDifficulty(9)
POS.debug.trigger('posBlackout')
POS.debug.tutorial('sale')
```

## 폴더 가이드

- `src/index.html`: 게임 엔트리 HTML
- `src/js/`: 게임 상태, 라운드 생성, UI, 튜토리얼, 오디오 로직
- `src/data/`: 상품, NPC, 프로모션, 응원 대사 데이터
- `src/assets/`: 배포용 SFX/BGM 및 보조 GIF
- `src/sound/`: 오디오 자산 재생성 스크립트
- `sandbox/`: 게임 본편과 분리된 실험용 화면
- `docs/`: 기획 드리프트, 디버그 명령, 개발 로그
